TAME itself is essentially a logic kernel that accepts input (usually submitted using a shell), processes it, changes a module context's state, and returns a result. Requests are created and sent to the context, and a Response is generated that contains data to relate to the user. This data is a list of items called Cues.

TAME's Request-Response model is similar to most other transaction-based systems. For example, web servers receive an HTTP Request from a web browser, find the correct Session, interpret the URL path, do something in the server code, and then return an HTTP Response with HTML to render in a browser. In this scenario, TAME is the server itself, the Module Context is the Session, the User Input is the URL, the Module is the package of files and server-side code, and the Shell is the web browser - the thing that submits requests and interprets responses, and the Cues are the "HTML".

2.1 Input and Requests

Input into TAME is usually a user-typed string like: 

pick up lamp
look at dog
scream
TAME takes this input and wraps it in a Request object. The Interpreter figures out the Action and the Objects involved and then queues that action for processing, then after all actions are processed, a Response is sent back about what happened in TAME as a result. Usually, this Response contains Cues that the shell needs to report to the user somehow.

2.2 Response and Cues

Every TAME request gets a Response. The Response contains things like:

  • How long it took to interpret the input.
  • How long it took to process the request.
  • How many commands called to complete the full request.
  • The list of Cues returned.
After the response is received, the Shell has to do something wih it. Each of the Cues in the Response is something for the Shell to do, in the order that they are read from the list, before it can send input again. Not every TAME Shell can or should handle these Cues, but there are a few types of Cues that ALL Shells need to handle.

You can see these Cues when you run modules in this documentation in DEBUG mode.

The fact that not every Shell can handle every Cue is the thing that gives TAME flexibility in authoring - some Cues can be graphical, some can change a screen layout, and some could refer to music or sound effects, but this is only relevant to special shells. If written properly, the same TAME module can be executed in both a rich multimedia environment, as well as a purely textual environment, like a command terminal or even across an IRC server!

2.3 Diagrams of Typical TAME Systems

For the most part, the most common TAME setup is this one - a user that interacts with a shell that changes a context, and the separation between shell, context, and module doesn't exist (for instance, most all-in-one adventures or games).

But to be honest, they do not have to be together conceptually - the shell can be a dumb terminal that relies on the context for parsing and execution, like a website that holds a context and the engine in memory, and can change its frontend appearance at will, or start new games.

In fact, in some multi-user versions, you can have a system built to handle input from many users with many separate contexts, all built from the same module, like a old-school Compuserve terminal system or an interactive IRC bot!

It is this separation that gives TAME additional flexibility, be it for authoring, rendering, or running on multiple platforms that have a TAME implementation. But regardless of the implementation, TAME still follows the same Request-Response model.
